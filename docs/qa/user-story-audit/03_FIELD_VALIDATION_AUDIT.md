# Field Validation Audit

This audit compares `docs/ba/04_FIELD_DICTIONARY.md`, `docs/ba/05_VALIDATION_RULES.md`, current backend schemas, and frontend forms.

Overall field validation verdict: Not Ready for full QA test generation.

Main reason:
The BA files identify most fields and many backend rules, but field-level validation is incomplete for max values, accepted characters, whitespace normalization, Arabic error messages, error placement, and frontend/backend consistency.

## Validation Gap Count

Total field validation and error-message gaps identified: 26.

## Backend/Frontend Validation Mismatches

| Area | Backend | Frontend | Risk |
|---|---|---|---|
| `serving_grams` | Optional, `> 0` if present | Number input appears to allow `0` | Backend rejection can feel unexpected. |
| Required nutrients | `>= 0` and required | Draft defaults to `0`; blank may become `0` | Missing data can be saved as real zero. |
| Text fields | `min_length=1` | HTML `required` | Whitespace-only values may pass. |
| Food duplicate | No constraint/check | No check | Duplicate catalog entries likely. |
| Net carbs | Computed as carb minus fiber | Display only | Negative net carbs possible. |
| Network/API errors | API returns errors or fetch rejects | Current mutation errors can write local data and queue sync | Violates online-only v1. |

## Required Validation Work Before QA Test Cases

1. Exact Arabic error messages for each field and form-level failure.
2. Max values for food nutrients, profile metrics, and diary quantity.
3. Accepted character policy for food name and serving label.
4. Duplicate food normalization rule using only confirmed v1 fields.
5. API error mapping rules for 401, 404, 422, timeout, and 5xx.
